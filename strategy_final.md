# strategy_final.md

## 1. このドキュメントの目的

`memo.md` にある「緊急修正」「将来機能」「設計上の疑問」を混在のまま進めると、手戻りが増える。
このドキュメントでは以下を整理する。

- 優先度（何を先に直す/作るべきか）
- 同時に固めるべき根幹設計（DB/RLS/API境界/公開範囲）
- 手戻りを減らす実装順
- 各フェーズの具体タスク
- テスト観点

前提として、現在のアーキテクチャ（`frontend` は Supabase 直接参照、`backend` は投稿/画像アップロード等の副作用処理）を活かしつつ、機能拡張に耐える形へ再整理する。

---

## 2. 全体方針（結論）

### プロダクト順序（価値提供の順）

`口コミ/授業情報 -> ノート/課題 -> 時間割 -> つながり（足跡/マッチング/DM） -> 収益化`

この順序は妥当。初期は「学習価値」を先に作り、ソーシャル/収益化は後段に置く。

### 技術順序（手戻り削減の順）

`基盤設計（大学スコープ/公開範囲/権限） -> データ取得基盤（home/community list API/RPC） -> MVP画面の実データ化 -> ソーシャル機能 -> 収益化`

理由:
- 公開範囲や大学スコープを後から変えると、RLS/UI/APIすべて修正になる
- DM/マッチングは安全設計（block/report/gating）が前提
- ホーム/時間割/授業・口コミを先に実データ化すると、利用ログ/KPIが先に取れる

## 3. 優先度整理（memo.mdベース）

## P0（今すぐ対応。既存UXの破綻/主要導線の不具合）

1. 授業・口コミの投稿時に「ログインが必要です」が出て投稿できない不具合修正
- 対象: ノート/口コミ/質問 投稿導線
- 影響: コア価値（UGC投稿）が止まる
- まずここを直さないと他の改善が無意味

2. ホーム画面のモックデータ脱却（`homeMockData` -> 実データ）
- 対象: ホームの一覧/ランキング/最近の投稿等
- 影響: 初回体験の信頼性
- 今後のKPI計測にも必須

3. マイページ設定「公開範囲保存」実装（UIだけで保存されない状態の解消）
- 対象: `SettingsPanel` の保存処理
- 影響: ユーザーのプライバシー制御
- 既にRLS方針があるなら、UI/DB接続を早く完了させるべき

## P1（P0と並走可。基盤不足でMVP価値が伸びない）

1. チャット/スレッド一覧API（またはRPC/view）整備
- 現状は作成RPCはあるが、一覧取得が不足
- コミュニティ機能の永続化前提になる

2. コミュニティのローカルフォールバック依存解消（非永続DMから脱却）
- 失敗時の救済としては良いが、常態化するとデータ整合性が崩れる

3. コミュニティのチップフィルタを実データクエリに反映
- UIだけ存在して実質無効は、誤認を生む

4. 時間割の主要プレースホルダ解消（検索/追加モーダル/受講者探索CTA）
- 時間割はマッチング基盤なので、最低限動く必要がある

## P2（価値拡張。基盤が固まった後）

1. DM scope 緩和（任意相手DM）
- 先に `dm_scope` / `allow_dm` / block/report / rate-limit 方針を固めてから

2. ノート投稿権限の見直し（`dropped` 講義でも投稿可にするか）
- プロダクト方針の決定が先（履修経験者として投稿可は自然）
- RLS/関数条件の変更で対応可能

3. マイページ「保存」タブ実装

4. コミュニティタブのプレースホルダ（reviews / more）実装

5. 人気授業ランキング
- ホーム実データ化後に着手（指標定義が必要）

6. 時間割セルの色選択
- UX向上として有効だが後回しでよい

## P3（新規事業/中長期）

1. 教科書交換機能
- 独立ドメイン。先に既存コア導線を安定化

2. 安全/収益化機能のUI接続（blocks / reports / profile_views / entitlements / subscriptions）
- 足跡はP2寄りだが、課金/購読UIはP3でもよい

3. リブランディング（ロゴ/LP/App導線）
- プロダクト価値検証とKPI改善後

## 4. 同時に固めるべき根幹設計（最重要）

ここを先に決めると、後工程の手戻りが大きく減る。

## 4.1 大学スコープ設計（大学ごとに見える範囲をどうするか）

### 推奨方針（初期）

- `profiles.university_id` をオンボーディング時に必須化
- 授業/口コミ/ノート/質問の一覧は「同大学」をデフォルト表示
- 将来のクロス大学は feature flag 的に後から拡張

### 理由

- 初期の relevance が高い（無関係な大学の授業が混ざるとUX悪化）
- RLS/クエリ条件が単純化する
- マッチング精度も上がる

### 設計ポイント

- `profiles` 未設定ユーザーへの初回導線（大学・学年入力）
- `notes/reviews/questions` の取得条件に `offering -> term -> university_id` を通す
- ホームAPI/RPCでも同様に大学スコープを統一

## 4.2 公開範囲設計（プロフィール/履修）

### 推奨方針

- `profiles` に本人プロフィール公開範囲の設定項目を持つ（将来拡張可能な enum）
- `enrollments` は行単位 `visibility` を維持しつつ、`profiles.enrollment_visibility_default` を初期値として使う
- マイページ設定の「一括変更」は以下の2段階で分離
  - デフォルト変更（今後の追加分に適用）
  - 既存履修の一括更新（明示実行）

### 理由

- ユーザー期待値に沿う（「今後」と「既存」が違う）
- 大量更新を勝手に走らせない設計にできる

## 4.3 API境界（frontend直叩き vs backend経由）

### 結論（ハイブリッド継続）

- `参照（read）`: フロントから Supabase 直接（RLS/RPC前提）
- `更新（write）` のうち単純なもの: Supabase（RLS + RPC）
- `副作用/複合処理/ファイル処理`: backend（Express）または Supabase Edge Function

### 判定基準

フロント直でよい:
- 単純SELECT
- RPCで安全に集計済みデータを返せる
- トランザクション不要

backend/RPCに寄せるべき:
- 複数テーブル更新が一体
- 権限判定が複雑
- 添付ファイル処理（サイズ/形式/メタデータ）
- 監査ログ/レート制限/通報判定を挟みたい

### ノート画像投稿への適用（設計指針）

- 既存 `multer` の再利用は可能だが、「assignments向け実装の流用」ではなくノート専用APIとして切る
- 推奨は `POST /api/notes`（multipart対応）または `POST /api/notes/assets` + `POST /api/notes`
- `notes` と `note_assets` を一貫した責務で扱う
- 将来の画像圧縮/NG判定/容量制限を見据えるなら backend 経由が無難

## 4.4 コミュニティ/DMの安全設計（先に最低限固める）

### 先に必要なもの

- `list_threads` 相当のRPC or view + query
- `can_dm()` / `dm_scope` / `allow_dm` の判定一貫化
- `blocks` / `reports` UI導線（最低限の入口）
- フロントのローカルフォールバックを「暫定機能」と明記し、常用しない

### 理由

- DMは不具合よりも安全/運用リスクが大きい
- 仕様が曖昧なままUI先行すると後で全部作り直しになる

## 4.5 重複授業（ユーザー追加講義）設計

### 問題

- 講義コードなしのユーザー作成講義で重複が起きると、マッチング精度/レビュー集約が崩れる

### 推奨方針（初期）

- `course_offerings` の直接乱立を避ける
- 追加時は「候補検索 -> 見つからなければ新規作成」
- 新規作成時に重複判定キー（大学/年度/学期/授業名正規化/教員名/曜日時限）で近似チェック
- 完全一致でなくても「既存候補を提示して選ばせる」UXにする

### 将来方針

- alias/統合テーブル or admin merge機能でデータクレンジング

## 5. 疑問への回答（設計判断）

## 5.1 他人から見えるプロフィール vs 自分プロフィール

### 推奨設計

- `My Profile`（本人向け）と `Public Profile`（他人向け）を分離して考える
- 本人向け: 全項目表示（編集可能）
- 他人向け: 公開設定に応じて表示項目を制限

### 実装方針

- フロントで分岐するだけでなく、取得クエリ/RPC側でも公開範囲を反映
- 「見えてはいけない項目」をUI非表示だけにしない（RLS/RPCで防ぐ）

### テスト観点

- A本人で見える
- B他人で見えない
- 同大学ユーザーなら見える項目
- ブロック関係時の表示制限

## 5.2 「2件投稿で解放」だと新入生が詰む問題

### 推奨方針

解放条件を単一条件にしない。

- `投稿2件以上` OR `時間割登録N件以上` OR `プロフィール充実度達成`

例（初期案）:
- 足跡詳細解放: `contributions >= 2 OR enrollments >= 3`
- DM送信解放: `contributions >= 2 OR review >= 1 + enrollments >= 3`

### 理由

- 新入生にも達成ルートがある
- スパム抑制を維持しつつオンボーディング詰まりを避ける

## 5.3 マッチング候補に「どの授業が被っているか」を出したい

### 推奨方針

- `find_match_candidates()` の返却は維持（集計中心）
- 追加で `shared_offering_preview` 系RPCを作る
  - 返すのは件数 + プレビュー（例: 最大3件の授業名）
  - 他者の履修全件を返さない

### 理由

- 現行のプライバシー方針（enrollments本人のみ参照）を崩さず改善できる

## 5.4 大学ごと表示を分けるべきか

### 推奨方針

- 初期は分ける（同大学デフォルト）
- オンボーディングで大学/学年を入力必須
- 将来のみ cross-university を検討

## 5.5 ユーザー追加講義の重複でマッチングが壊れないか

### 結論

- 現状のままだと壊れる/精度低下のリスクが高い
- 上記の重複判定フロー（検索優先 + 候補提示 + 近似チェック）を先に導入すべき

## 6. 手戻りを減らす実装順（推奨ロードマップ）

## Phase 0: 安定化（1〜2週間）

目的: 既存の壊れている導線を直し、実データ運用の土台を作る

1. 投稿系ログイン不具合修正（ノート/口コミ/質問）
2. マイページ設定「公開範囲保存」実装
3. ホーム実データ化（最低限）
4. コミュニティ/時間割のプレースホルダ棚卸し（どこまでMVPで動かすか確定）

成果物:
- 投稿できる
- 設定が保存される
- ホームが実データで表示される

## Phase 1: 基盤確定（1〜2週間）

目的: 後から壊しにくいルールを先に固定する

1. 大学スコープ方針の実装（オンボーディング + クエリ統一）
2. 公開範囲モデルの整理（profiles/enrollmentsの役割分担）
3. 重複講義追加フロー設計（検索優先 + 新規作成）
4. コミュニティ一覧用 RPC/view（`list_threads`相当）追加設計
5. `shared_offering_preview` RPC設計（マッチング説明用）

成果物:
- データスコープが統一される
- マッチング/DMの前提APIが揃う

## Phase 2: 学習価値MVPの完成（2〜4週間）

目的: 「便利だから使う」を成立させる

1. ラクタン口コミページの完成度向上（一覧/詳細/投稿導線）
2. 時間割ページMVP（検索/追加/一覧表示/編集最低限）
3. プロフィール充実（学部・学年・興味）
4. ホームに人気授業/最近のノート等を反映（軽量版）

成果物:
- 口コミ -> 時間割 -> ノートの導線が機能
- 新規ユーザーがプロフィール/時間割まで到達できる

## Phase 3: ソーシャルMVP（2〜4週間）

目的: 継続率を上げる

1. 足跡（軽量版） + 解放条件
2. マッチングページ（共有授業プレビュー付き）
3. スレッド一覧/DM送受信の永続化
4. `dm_scope` / `allow_dm` / block/report の最低限UI
5. ローカルフォールバック縮小（障害時のみ）

成果物:
- マッチング理由が説明できる
- DMが永続化される
- 安全導線が最低限ある

## Phase 4: 収益化/運用強化（後段）

1. `entitlements` / `subscriptions` UI接続
2. 機能解放の課金導線
3. 通報/ブロック運用オペレーション整備
4. LP/リブランディング導線

## 8. 実装ルール（このプロジェクト向け）

## 8.1 参照/更新の責務

- 参照:
  - 基本は frontend -> Supabase（RLS/RPC）
  - 他者の個人情報に触れるものは raw table を避け、RPCで集計/公開範囲適用

- 更新:
  - 単純更新は Supabase 直（RLSで守る）
  - 複合更新/添付/監査は backend API

## 8.2 SQL/RLS先行の原則（コミュニティ/DM系）

- UIを先に作り切らない
- 先に以下を確定
  - 判定関数（`can_dm`, `can_view_footprints`）
  - 一覧RPC（`list_threads`）
  - 安全導線（block/report）

## 8.3 マイグレーション運用

- このworktreeでは `supabase db reset` / `supabase migration up` を自動実行しない
- DB変更が必要な場合は `supabase migration new` でマイグレーションファイルのみ作成
- 変更内容は `docs/db_schema.md` / `docs/data-model.md` も追随更新

## 9. テスト戦略（最低限）

## 9.1 優先テスト（不具合防止）

1. 認証あり/なしで投稿導線の挙動
- 未ログイン: ログイン誘導
- ログイン済み: 投稿フォーム送信成功
- セッション期限切れ: 再ログイン誘導

2. 公開範囲テスト
- 自分から見えるプロフィール
- 他人から見えるプロフィール
- `enrollment_visibility_default` と既存履修一括更新の挙動差

3. コミュニティDM権限テスト
- `dm_scope` / `allow_dm` / ブロック時の拒否
- 解放条件未達時の拒否

4. 重複講義追加フロー
- 既存授業ヒット時に新規作成しない
- 近似候補提示が出る

## 9.2 KPI計測の最低限（将来の判断材料）

- ホーム閲覧率
- 口コミ閲覧率
- 口コミ/ノート投稿率
- 時間割登録率
- マッチング候補閲覧率
- DM開始率（Phase 3以降）

## 10. 最終的な優先順位（実務用・短く）

1. 壊れているコア導線を直す（投稿ログイン不具合 / 公開範囲保存）
2. ホームを実データ化して初回体験を本番化
3. 大学スコープ・公開範囲・重複講義の基盤を固める
4. 時間割/口コミMVPを完成させる

6. 収益化・教科書交換・リブランディングは後段

## 11. 補足（memo内の「解決済み」項目の扱い）

`memo.md` 内で「解決済み」とある項目は、基本的に本ドキュメントの優先タスクから外す。
ただし回帰防止のため、関連箇所を触るタイミングで最小限の確認は行う。
