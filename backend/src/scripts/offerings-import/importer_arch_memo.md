# 専修大学シラバス importer の設計変更メモ

## 1. 背景

専修大学シラバスから講義データを取得し、アプリの `offering` 系データへ取り込む importer を Playwright ベースで実装していた。

当初の目的は以下だった。

* 指定年度・指定学期の講義を専修大学シラバスから取得する
* `offering` / `course` / `slot` に正規化して投入できる形へ変換する
* `source_mappings` を通じて冪等に再 import できるようにする
* 将来的には専修大学の講義をある程度まとまった件数で事前投入し、新規ユーザーがすぐ時間割登録できるようにする

ただし、実装を進める中で、対象サイトの UI・セッション管理・件数規模に起因する複数の問題が見つかり、設計方針を段階的に見直した。

---

## 2. 最初の実装方針

最初の importer は、おおむね以下の流れだった。

1. 検索画面にアクセスする
2. 年度・学部等・学期を選択して検索する
3. 結果一覧から詳細ページ URL を集める
4. 集めた各詳細 URL に対して `page.goto(detailUrl)` で直接アクセスする
5. 詳細本文をテキストとして読み、`courseTitle` / `instructor` / `termCode` / `slots` などをパースする
6. `CanonicalOfferingImportItem[]` を返す

この方式の利点は、

* 一覧収集と詳細取得を責務分離しやすい
* 詳細 URL が集まれば後続処理を単純に回せる
* 一覧ページと詳細ページを分けて考えられる

という点だった。

一方で、実サイトに適用すると、次の問題が見えた。

---

## 3. 件数規模の問題が先に見えた

まず分かったのは、2025 年度の全件取得を素直に回すと、検索結果が数千件規模になるということだった。

観測上、2025 全件では 9000 件超が存在しており、当初の「全件をひとまず全部取る」という方針は、次の理由で現実的ではないと判断した。

* Playwright の巡回に非常に時間がかかる
* サイト側の安定性に依存しすぎる
* import 時間が長くなりすぎる
* 本番投入前の検証粒度が粗すぎる
* アプリ側でも、大量 `offering` 前提の検索・表示設計が未確定だった

この段階で、まず import のスコープを狭める方針に変えた。

---

## 4. import スコープを「全件」から「限定対象」へ切り替えた理由

### 4.1 なぜ全件投入をいったんやめたか

DB 的には `offering` が 1 万件前後あること自体は大きな問題ではない。

しかし、アプリ全体では次のような悪影響が考えられた。

* 時間割追加画面で全件を一度に出すと UI が破綻する
* 絞り込みなし検索だと API が重くなる
* 学期・学部単位の UX が未整理のまま件数だけ増える
* import ロジックの正当性検証がしづらい

そのため、まずは

* 特定年度
* 特定学期
* 特定学部（またはその配下の学科群）

に限定して importer を回し、import ロジックそのものを安定化させる方針に変更した。

### 4.2 `ImportScope` の拡張

その流れで CLI 側に次のような指定を持たせた。

* `--academic-year`
* `--term`
* `--department`
* `--dry-run`
* `--retire-missing`

ここで特に重要だったのは `--department` で、これにより「まずは経済学部だけ」といった限定 import をできるようにした。

---

## 5. `開講学部等` の解釈を修正した

### 5.1 最初の誤解

当初は、`value(crclm)` の選択肢を「学部」と思っていた。

しかし実際に UI を確認すると、ドロップダウンは次のような構造になっていた。

* `【経済学部】`
* `経済学科`
* `現代経済学科`
* `生活環境経済学科`
* `国際経済学科`
* `二部経済学科`
* `【法学部】`
* `法律学科`
* `政治学科`
* ...

つまり、`【経済学部】` のようなものは見出しであり、実際に検索条件として選ばれるのはその配下の学科だった。

### 5.2 変更したこと

そこで `selectDepartmentLabels()` を導入し、次のルールで解釈するようにした。

* `--department "経済学部"` のように見出し名を渡した場合

  * `【経済学部】` の次から、次の見出しが出るまでの項目を全部選ぶ
* `--department "経済学科"` のように個別名を渡した場合

  * その項目だけ選ぶ

### 5.3 なぜそうしたか

これは実 UI 構造に合わせるためであり、ユーザー意図としても

* 「学部単位で取りたい」
* しかし実サイトの検索は「学科単位でしかできない」

というギャップを吸収するためだった。

---

## 6. 結果一覧のページング方式の誤認を修正した

### 6.1 最初の想定

最初は検索結果が通常のページネーション、つまり

* 1 ページ目
* 2 ページ目
* 次へ

のような構造だと考えていた。

そのため、`collectDetailLinks()` でも「`a` タグの `次` を探してクリックする」という設計を採っていた。

### 6.2 実サイトで分かったこと

実際の結果画面を見ると、右上に `検索結果 1491 件` のような総件数表示があり、下部には

* `次の5件を読み込む`

というボタンが存在していた。

DOM 上でも、これは通常のページネーションリンクではなく、`input type="submit"` として描画されていた。

### 6.3 変更したこと

そこで `collectDetailLinks()` を修正し、

* `a` の「次」リンクを探す方式

ではなく、

* `input[type="submit"][value*="次の5件"]`
* `input[type="submit"][value*="読み込む"]`

を押して、見えている詳細リンク数が増えるまで待つ方式に変更した。

### 6.4 なぜそうしたか

理由は単純で、実サイトが「ページ移動」ではなく「同一画面上の追加ロード」型だったためである。

この変更により、最初の 5 件だけではなく、少なくとも 10 件、15 件と追加ロードできるようになった。

---

## 7. それでも詳細 URL 直アクセスが失敗した

### 7.1 起きた現象

結果一覧のリンク数回収はある程度改善したが、その後に詳細 URL へ `page.goto(detailUrl)` で直接アクセスすると、60 秒待っても `domcontentloaded` に到達せずタイムアウトする問題が起きた。

実際のログでは、たとえば

* 検索結果のリンク収集までは成功
* 1 件目の詳細 URL へ移動した瞬間にタイムアウト

という形だった。

### 7.2 切り分け

この時点で考えられた原因は主に 3 つあった。

1. セッション依存
2. サーバー側のレート制限
3. 待機条件の不適合

その後、シークレットウィンドウで詳細 URL を直接開いたところエラーになったため、少なくとも「詳細 URL 単体では成立しない」ことが確認できた。

### 7.3 結論

このサイトは、

* 正規の検索画面を通り
* 結果一覧から遷移したセッション状態

を前提として詳細画面を返している可能性が高いと判断した。

つまり、「一覧で URL だけ集めておき、あとで別ページから URL 直打ちで取得する」方式が、このサイトでは成立しなかった。

---

## 8. 設計を「URL 収集型」から「正規クリック遷移型」へ変更した

### 8.1 変更前の設計

変更前は、

* 検索する
* 一覧から詳細 URL を集める
* 別ページで `goto(detailUrl)` する

という構造だった。

### 8.2 変更後の設計

変更後は、

* 検索する
* `次の5件を読み込む` を必要なところまで押して結果一覧を展開する
* 一覧上の `a[href*="slspsbdr.do"]` を index 順にクリックする
* 詳細ページ本文を読み取る
* `goBack()` で一覧へ戻る
* 次の index に進む

という方式に切り替えた。

### 8.3 なぜそうしたか

この方式にした理由は、サイトの正規ナビゲーションフローを守るためである。

* 検索結果ページからクリックして遷移する
* 詳細を見たら戻る

という、人間の利用フローに近い動きをそのまま再現することで、セッション依存の問題を回避しやすくした。

---

## 9. 戻った後に一覧状態が失われる問題も考慮した

### 9.1 問題

一覧から 1 件クリックして詳細へ行き、`goBack()` で戻ると、すでに読み込んでいた件数が保持されない可能性がある。

たとえば 50 件分ロード済みの状態で 37 件目を開いたとしても、戻った後に再び最初の 5 件しか見えなくなることがありうる。

### 9.2 変更したこと

そこで、`ensureResultsLoadedToIndex(page, targetIndex)` のような責務を持たせ、

* 今から index 番目を押したい
* その index が画面上に見えるところまで `次の5件を読み込む` を繰り返す

という処理を導入した。

### 9.3 なぜそうしたか

これにより、戻った後に一覧状態が崩れても、「次に押したい要素が見える状態」まで復元できるため、順次巡回を継続しやすくなる。

---

## 10. 件数把握ログを増やした

### 10.1 変更したこと

結果画面にある `検索結果 1491件` のような件数を `readSearchResultCount()` で読むようにした。

そのうえでログに

* expected: 画面上の総件数
* extracted / visible: 実際に取得できた件数

を出すようにした。

### 10.2 なぜそうしたか

この変更には 2 つの意味がある。

1. 実際にどの程度取りこぼしているかを把握できる
2. ロジック修正後に、改善の度合いを件数ベースで検証できる

「経済学科 前期で 715 件あるはずなのに 15 件しか取れていない」といった状況がすぐ分かるようになり、問題の切り分けがしやすくなった。

---

## 11. dry-run を維持した理由

### 11.1 変更前後を通じて維持したこと

CLI 側の `--dry-run` は引き続き維持した。

このモードでは

* スクレイピングとパースは行う
* ただし DB の upsert は行わない

という動きになる。

### 11.2 なぜそうしたか

今回のように、

* UI の構造理解
* セッション問題の切り分け
* 巡回ロジックの調整

が主目的の段階では、DB へ実際に反映しない方が安全だからである。

import ロジックの安定化前に実テーブルを汚さないため、dry-run は重要だった。

---

## 12. 現時点の importer の考え方

現時点での設計思想は、次のように整理できる。

### 12.1 スコープは最初から狭く取る

* 年度限定
* 学期限定
* 学部または学科限定

として、小さな単位で importer を安定化させる。

### 12.2 専修シラバスの UI 構造を尊重する

* `開講学部等` は見出しと学科群の階層構造を持つ
* 結果一覧は「次の5件を読み込む」型である
* 詳細ページは URL 直アクセスではなく、正規遷移が前提である

### 12.3 データ件数よりも UX と運用を優先する

いきなり全件投入するのではなく、

* importer が安定しているか
* アプリ側の検索 UX が成立しているか
* 年度・学期・学部ごとの絞り込みが整っているか

を優先して進める。

---

## 13. 今後の改善候補

現時点で考えられる今後の改善候補は以下である。

### 13.1 巡回の安定性改善

* `goBack()` 後の復元が本当に安定しているか検証する
* click 後の待機条件をさらに厳密にする
* 詳細ページで本文抽出前に必要な要素待機を追加する

### 13.2 パフォーマンス改善

現方式は、件数が多いとどうしても遅い。

そのため、将来的には

* 学部単位でのバッチ実行
* 差分更新
* 人気講義や対象学部だけの優先投入

などを検討する余地がある。

### 13.3 アプリ側 UX との接続

大量 `offering` を前提にする場合は、アプリ側でも

* 年度学期で絞る
* 学部で絞る
* キーワード検索を前提にする
* 一覧全件表示を避ける

といった設計が必要になる。

---

## 14. まとめ

今回の設計変更を一言でまとめると、

**「専修大学シラバスの実際の UI とセッション制約に合わせて、importer を URL 直アクセス型から正規クリック遷移型へ変更した」**

となる。

具体的な変化は次の通りである。

* 全件 import 前提をやめ、限定スコープで検証する方針にした
* `開講学部等` を「見出し + 学科群」として扱うようにした
* 結果一覧の取得を「次へ」前提から「次の5件を読み込む」前提に変えた
* 詳細 URL の直接 `goto()` をやめ、一覧からの正規クリック遷移に変えた
* `goBack()` 後の一覧復元を考慮した
* 総件数ログを出して、取りこぼしを見える化した
* dry-run で DB を汚さずに検証できる形を維持した

この一連の変更により、専修大学シラバスというレガシーかつセッション依存の強いサイトに対して、より現実的で安定した importer の形へ近づけた。
